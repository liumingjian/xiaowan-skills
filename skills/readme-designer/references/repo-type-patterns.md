# Repo Type Patterns

在写 README 前，先识别仓库主类型。不要为了“全面”同时选多个主类型。

## Type Signals

### `library/framework`

常见信号：

- 有明确公开 API、SDK、hooks、组件、库入口
- 核心价值是“被别的项目引入或调用”
- 文档重点是安装、最小示例、API 入口

首屏优先顺序：

1. 项目简介
2. 适用 / 不适用场景
3. 安装
4. 最小示例
5. 核心能力
6. 文档导航

### `cli/tool`

常见信号：

- 有命令入口、子命令、flags、配置文件
- 核心价值是直接在终端里完成任务
- 文档重点是安装、真实命令、输出结果、配置

首屏优先顺序：

1. 项目简介
2. 适用 / 不适用场景
3. 安装
4. 最小命令示例
5. 常用任务
6. 配置说明

### `app/service`

常见信号：

- 有服务端或前端运行入口
- 用户通过 UI、HTTP 服务、控制台页面或后台系统使用
- 文档重点是功能概览、先决条件、启动、访问方式、部署

首屏优先顺序：

1. 项目简介
2. 截图或效果概览（可选）
3. 适用 / 不适用场景
4. 先决条件
5. 快速开始
6. 核心功能

### `template/starter`

常见信号：

- 核心价值是“生成一个新项目”
- 有脚手架、初始化命令、模板变量、生成后的目录
- 文档重点是创建方式、生成结果、下一步

首屏优先顺序：

1. 项目简介
2. 适用场景
3. 创建命令
4. 生成后包含什么
5. 下一步
6. 文档导航

## Maturity Signals

### `early`

信号：

- 还没有稳定发布或 release
- 文档、测试、CI 或贡献规范不完整
- 核心能力可运行，但边界多

写法：

- 只承诺当前真实可用路径
- 强调限制和前置条件
- 不写稳定性或生产级承诺

### `growing`

信号：

- 主路径稳定
- 有基本文档、示例、测试或 CI
- 仓库已有外部使用可能

写法：

- 强调快速开始、常见任务、文档导航
- 可写“推荐使用方式”，不要夸大生态

### `mature`

信号：

- 发布、文档、贡献规范较完整
- 有稳定入口、示例和治理信息
- 用户可预期长期使用

写法：

- 可以增加兼容性、生态、治理和贡献信息
- 仍然保持首屏简洁，不要把全部信息堆回 README

## Visual Guidance

- `app/service`：优先截图、GIF 或简图
- `cli/tool`：优先真实命令与结果，截图通常次要
- `library/framework`：优先代码示例，通常不需要截图
- `template/starter`：优先初始化命令和生成结果说明

## Ambiguity Rule

如果仓库既像 `cli` 又像 `library`，按“第一次进入仓库的外部使用者最先要完成什么”决定主类型：

- 先安装并调用代码：偏 `library/framework`
- 先执行命令完成任务：偏 `cli/tool`

不要为了求稳把 README 写成两套入口并列，那会破坏首屏清晰度。
