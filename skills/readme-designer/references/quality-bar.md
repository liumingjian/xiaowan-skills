# README Quality Bar

在交付 README 前，用下面的规则自检。

## Scoring

每项 0 到 2 分，总分 12 分。低于 9 分不要直接交付。

### 1. 清晰度

- 0：开头读完仍不知道项目是什么
- 1：知道大概是什么，但定位含糊
- 2：首屏立刻回答“这是什么、解决什么问题”

### 2. 上手性

- 0：只有安装，没有成功路径
- 1：有步骤，但成功标志不清楚
- 2：能顺着文档完成第一次成功结果

### 3. 信息架构

- 0：章节堆叠，没有主次
- 1：结构基本合理，但顺序不够贴合仓库类型
- 2：首屏顺序清楚，正文深度合适

### 4. 真实性

- 0：存在臆造命令、能力或承诺
- 1：基本真实，但有几句无依据判断
- 2：所有关键结论都能回到仓库事实

### 5. 边界感

- 0：只写功能，不写限制
- 1：写了少量限制，但不够实用
- 2：适用、不适用、限制、前置条件都清楚

### 6. 导航性

- 0：README 像杂乱笔记
- 1：有文档链接，但没有分流逻辑
- 2：能把不同读者快速导向更深资料

## Hard Rules

交付前必须满足：

- 首屏不堆长徽章墙
- `快速开始` 不是纯安装说明
- 不把大量配置表塞进 README
- 不保留空章节、模板占位或“TODO”痕迹
- 不写“强大、现代化、企业级、优雅、高性能”这类无证据形容
- 已有 `README.md` 时，先给重写大纲并获得确认，再覆盖根文件
- 覆盖现有 `README.md` 时，先备份旧版到 `docs/`

## Anti-Patterns

出现以下情况时，优先重写结构而不是修句子：

- README 一上来就是项目背景长文
- 功能列表直接等于源码模块列表
- 所有命令都放在一个大代码块里，没有说明顺序和成功标志
- 架构、目录、配置、贡献、FAQ 全部平铺展开
- 旧 README 明显是面向维护者，不是面向首次读者

## Missing Facts

信息不足时，允许交付高质量部分，但必须显式列出缺失项，例如：

- 许可证未确认
- 部署方式未确认
- 截图资源缺失
- 兼容版本未确认

不要为了显得完整而填补假信息。
