# README Designer

## Overview

为仓库生成专业、克制、可快速上手的中文 README。默认面向第一次进入仓库的外部开发者，优先产出导航型 README，而不是大而全说明书。

## Workflow

按以下顺序执行，不要跳步：

1. 确认任务边界。
2. 读取真实仓库事实。
3. 推断仓库类型与成熟度。
4. 先给重写大纲。
5. 用户确认后再重写 README。
6. 自检并暴露缺口。

## 1. Confirm Scope

仅在用户明确要求处理 `README.md` 时使用本 skill。普通 `docs/` 文档、接口文档、手册类任务不要自动套用。

固定边界：

- 主读者：第一次进入仓库的外部使用者
- 默认风格：开源展示型、中文、导航优先
- 默认策略：结构性重写优先于零碎修补
- 默认产物：先给重写大纲；只有在用户确认后才改写根目录 `README.md`

默认写入路径规则：

- 已有 `README.md`：先输出重写大纲，不要默认落盘为草稿文件
- 用户确认重写后：先备份旧版 `README.md` 到 `docs/readme-backup-YYYYMMDD.md`，再直接重写根目录 `README.md`
- 仓库没有 `README.md`：在提纲确认后直接创建根目录 `README.md`
- 用户只要方案或审阅意见：不要写文件

硬规则：

- 只要仓库里已经存在 `README.md`，未经用户确认不要直接重写
- 也不要用 `docs/readme-draft.md` 代替这个确认步骤
- 备份是重写现有 `README.md` 的默认动作，不需要额外询问

## 2. Read Real Repo Facts

先读事实，再写结论。至少检查以下信息源中实际存在的那些：

- 根目录 `README.md`
- `package.json`、`pyproject.toml`、`Cargo.toml`、`go.mod`、`Makefile`
- `Dockerfile`、`docker-compose.yml`、部署或运行脚本
- 主要源码入口、目录结构、公开 API 或 CLI 入口
- `docs/`、`examples/`、`test/`、演示数据、截图资源
- `LICENSE`、`CONTRIBUTING.md`

提取这些信息：

- 项目是什么，解决什么问题
- 真实可执行的安装、启动、构建、测试命令
- 首次成功结果的可观察标志
- 用户能完成的关键任务
- 必需配置、常用可选配置、外部依赖
- 明确限制、前置条件、不适用场景

硬规则：

- 不要生成仓库里不存在的命令、脚本名、目录名或环境变量
- 不要继承旧 README 中未经验证的宣传性说法
- 当旧 README 结构差但事实仍然有效时，只保留事实，不保留表达方式

## 3. Infer Type And Maturity

先判断仓库主类型，再判断成熟度。

主类型限定为：

- `library/framework`
- `cli/tool`
- `app/service`
- `template/starter`

成熟度限定为：

- `early`
- `growing`
- `mature`

如果需要类型判断信号、首屏顺序或类型差异，读取 [references/repo-type-patterns.md](references/repo-type-patterns.md)。

判断规则：

- 给出一个主类型，不要同时给多个
- 若仓库跨类型，按“第一次使用者最先要完成的路径”归类
- 若定位存在实质歧义，再给 2 个一句话定位候选；否则只给 1 个最优版本
- 成熟度只影响承诺强度、章节深度和边界表达，不改变基本骨架

## 4. Outline Before Rewrite

在改写任何现有 `README.md` 之前，先给一个 5 到 10 行的重写大纲，至少包含：

- 一句话项目定位
- 仓库类型与成熟度判断
- README 首屏要回答的 5 个问题
- 准备保留在 README 的 section
- 准备下沉到 `docs/` 或其他文件的内容

默认不要展开成长方案文档。这个大纲用于让用户确认定位、边界和重写方向。

只有在以下情况之一时，才跳过确认并直接写文件：

- 仓库里原本没有 `README.md`
- 用户明确要求“直接重写”“直接覆盖”“不要再确认”

## 5. Rewrite README

确认通过后再写文件。

当仓库已有 `README.md` 时，执行顺序固定为：

1. 备份旧版到 `docs/readme-backup-YYYYMMDD.md`
2. 直接重写根目录 `README.md`
3. 不再额外生成 `docs/readme-draft.md`

默认使用中等完整度 README：首屏短，正文稳，复杂细节下沉。

默认骨架如下，按仓库类型微调顺序与深度：

1. 项目简介
2. 适用场景 / 不适用场景
3. 先决条件
4. 快速开始
5. 核心功能
6. 配置说明
7. 项目结构或工作原理
8. 文档导航
9. 贡献与许可

强制要求：

- `项目简介` 用两句写法：是什么；解决什么问题/适合谁
- `快速开始` 必须走到第一个成功结果，不能只停在安装
- `配置说明` 必须区分“必需配置”和“常用可选配置”
- `核心功能` 按用户任务写，不按内部技术点堆砌
- 必须写限制、边界和不适用场景
- README 只做入口，不展开所有实现细节

写作风格：

- 中文
- 技术文档腔
- 先结论后解释
- 短句
- 少形容词
- 少口号

默认不要写这些表达：

- “强大而灵活”
- “现代化解决方案”
- “企业级能力”
- “高性能/高可扩展/易维护”，除非仓库里有明确证据

复杂内容的处理规则：

- 详细配置：下沉到 `docs/configuration.md`
- 详细架构：下沉到 `docs/architecture.md`
- 详细部署：下沉到 `docs/deployment.md`
- 详细贡献流程：下沉到 `CONTRIBUTING.md`

只有仓库已经存在这些文档时才直接链接；否则在交付说明中给补充建议，不要自动编造整套文档。

## 6. Self-Review

完成重写前，读取 [references/quality-bar.md](references/quality-bar.md) 并自检。

最低要求：

- 首屏能在 20 秒内看懂项目定位与上手入口
- 用户能在 5 分钟内完成项目判断
- 用户能在 15 分钟内完成首次上手
- 没有无依据断言
- 没有空章节和模板痕迹

如果评分低于合格线，先重写再交付，不要把低质量 README 直接交给用户。

## Output Contract

默认输出按阶段区分：

### 阶段 1：已有 README，等待确认

1. 极短设计摘要
2. 重写大纲
3. 缺失事实与假设
4. 建议下沉的配套文档

### 阶段 2：确认后执行重写

1. 重写后的 `README.md`
2. 旧版备份路径
3. 缺失事实与假设
4. 建议下沉的配套文档

### 阶段 3：仓库没有 README

1. 极短设计摘要
2. 新建的 `README.md`
3. 缺失事实与假设
4. 建议下沉的配套文档

如果用户只要求评审现有 README：

- 先给问题清单
- 再给结构性改进建议
- 未经要求不要直接重写整篇
